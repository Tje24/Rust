1. BORROWING CONFLICT EN EDITOR_APP.RS
---------------------------------------
Ubicación: src/core/editor_app.rs (implementación de eframe::App)
Problema: Múltiples préstamos mutables de `self` dentro del closure de egui

Código problemático:
```rust
egui::SidePanel::left("files_panel").show(ctx, |ui| {
    self.ui.draw_files_panel(ui, &mut self.mode, &mut self.viewport_mode, &self.project_path);
    //                    ^^                                ^^^^^^^^^^^^^^^^^^^
    //                    |                                 |
    //                    |                                 immutable borrow
    //                    mutable borrow of self.ui
});
```

Solución aplicada: Separar los datos necesarios y pasar solo referencias específicas.

3. DEPENDENCIAS CIRCULARES POTENCIALES
--------------------------------------
Riesgo: Los módulos scripting, ecs, y core tienen referencias cruzadas

Flujo actual:
- core::EditorApp usa scripting::Scene
- scripting::Scene usa scripting::Node
- scripting::Node usa scripting::components::Component
- ecs::Entity es independiente (HashMap con trait objects)

Recomendación: Mantener esta separación está bien, pero evitar que ecs 
dependa de scripting.

--------------------------------------------------------------------------------

4. TRAIT OBJECTS NO CLONEABLES
------------------------------
Ubicación: src/ecs/entity.rs
```rust
pub components: HashMap<String, Box<dyn std::any::Any + Send + Sync>>
```

Problema: Entity no puede derivar Clone porque los trait objects no son cloneables.

Solución aplicada: Eliminar Clone del derive, solo Debug.

--------------------------------------------------------------------------------

5. ANDROID ACTIVITY API CAMBIANTE
---------------------------------
Ubicación: src/lib.rs
Problema: android_activity 0.6.x cambió la API de poll_events()

API antigua (0.5.x):
```rust
app.poll_events() // Retorna Option<Event>
```

API nueva (0.6.x):
```rust
app.poll_events(timeout, callback) // Retorna (), usa callback
```

Solución aplicada: Usar callback con timeout.

7. CLOSURES EN HASHMAP
----------------------
Ubicación: src/scripting/behavior.rs

Problema: Cada closure tiene tipo único en Rust

Código incorrecto:
```rust
registry.insert("oscillate", Box::new(|| ...));
registry.insert("patrol", Box::new(|| ...));
// Error: tipos de closure diferentes
```

Solución aplicada: Usar function pointers (fn()) en lugar de Box<dyn Fn()>

```rust
pub type BehaviorRegistry = HashMap<String, fn() -> Box<dyn Behavior>>;
```

RECOMENDACIONES DE MEJORA

8. SEPARAR LÓGICA DE UI
-----------------------
Problema: EditorUI tiene demasiadas responsabilidades

Recomendación:
- Separar draw_files_panel en su propio módulo
- Separar draw_inspector en su propio módulo
- Usar traits para panels intercambiables

--------------------------------------------------------------------------------

9. GESTIÓN DE ERRORES
---------------------
Problema: Muchos unwrap() y expect() implícitos

Recomendación:
- Usar Result<T, E> en funciones que pueden fallar
- Implementar errores personalizados con thiserror
- Propagar errores con ? en lugar de unwrap()

--------------------------------------------------------------------------------

10. TESTS FALTANTES
-------------------
Archivos sin tests:
- src/core/editor_app.rs
- src/scripting/scene.rs
- src/ecs/world.rs

Recomendación:
- Agregar tests unitarios por módulo
- Usar #[cfg(test)] modules
- Implementar integration tests en /tests

--------------------------------------------------------------------------------

11. DOCUMENTACIÓN
-----------------
Problema: Pocos comentarios de documentación pública

Recomendación:
- Agregar //! doc comments para cada módulo
- Agregar /// para funciones públicas
- Usar #[example] en funciones complejas
